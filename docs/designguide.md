# 从 Skynet 学 Go 游戏服务器架构（第一阶段总结）

> 学习范围：Skynet 早期设计思想与架构演进（基础篇）

## 核心结论

Skynet 最值得学习的不是 Lua，而是 **Actor + 消息驱动 + 清晰职责边界**
的设计思想。这些思想可以直接迁移到 Go 游戏服务器开发。

------------------------------------------------------------------------

## 1. Actor 是状态唯一拥有者

-   每个 Actor 独占自己的可变状态。
-   Actor 之间只通过消息通信。
-   避免共享内存和大量锁。

推荐拆分：

-   GateActor：连接管理
-   RoomActor：分组、座位、重连
-   AttackActor：战斗时间线、规则校验
-   PlayerActor：玩家个人状态

------------------------------------------------------------------------

## 2. 消息驱动

推荐统一消息语义：

-   Tell()：异步通知
-   Ask()：请求/响应
-   Reply()：回复
-   Publish()：广播
-   Dispatch()：路由

不要直接跨 Actor 修改状态。

------------------------------------------------------------------------

## 3. API 保持简单

遵循 Skynet 风格：

-   API 数量少
-   一个动词只表达一种行为
-   不暴露底层实现

推荐：

``` go
Start()
Stop()
Tell()
Ask()
Reply()
Register()
Lookup()
Spawn()
Close()
```

避免：

``` go
ExecutePlayerOperation()
HandleBattleLogic()
SendMessageAsyncWithTimeout()
```

------------------------------------------------------------------------

## 4. 职责边界清晰

Gate：

-   网络连接
-   收发包
-   路由

Room：

-   玩家分组
-   座位
-   重连

Attack：

-   时间线
-   怪物生成
-   命中校验
-   广播

Player：

-   金币
-   血量
-   Buff
-   结算

------------------------------------------------------------------------

## 5. 生命周期统一

建议统一：

-   New()
-   Init()
-   Start()
-   Stop()
-   Close()

避免一个项目混用 Boot、Launch、Destroy、Dispose 等命名。

------------------------------------------------------------------------

## 6. 并发原则

-   一个 Actor 一个 Mailbox。
-   Actor 内尽量无需 Mutex。
-   长耗时任务不要阻塞 Actor Loop。
-   Ask 必须支持超时。

------------------------------------------------------------------------

## 7. 标识设计

推荐：

-   player_id
-   session_id
-   actor_id
-   attack_id
-   attack_epoch
-   spawn_seq
-   timeline_rev

利用 epoch 和 revision 防止旧消息误处理。

------------------------------------------------------------------------

## 8. 可观测性

建议内置：

-   Mailbox 长度
-   消息处理耗时
-   Ask 延迟
-   Socket 写队列
-   Trace ID
-   Actor ID

------------------------------------------------------------------------

## 9. 常见反模式

避免：

-   GameManager 包含所有业务
-   RoomActor 修改 Player 状态
-   Gate 执行业务逻辑
-   大量共享状态
-   API 命名混乱

------------------------------------------------------------------------

## 推荐目录

``` text
actor/
gate/
room/
attack/
player/
protocol/
```

## 总结

Skynet 最值得 Go 开发者学习的是：

1.  Actor 拥有状态
2.  消息驱动代替共享状态
3.  API 简洁且稳定
4.  明确职责边界
5.  生命周期统一
6.  良好的可观测性

这些原则非常适合 Go + protobuf + Actor
架构的棋牌游戏、打地鼠及其他多人在线游戏服务器。
